# AI 配将模块

`src/team-ai.js` 是独立的配将推荐模块，目标是让郊野守军、玩家自动整备和以后新增的推荐配将入口复用同一套逻辑。

## 当前接入点

- `randomEnemyTeam()`：郊野守军每次刷新时调用 `buildEnemyTeam()`。
- `autoTeam()`：玩家点击自动整备时调用 `recommendTeam()`，只使用当前已拥有武将和已解锁战法。
- 页面入口 `index.html` 在 `team-ai.js` 之前加载 `src/skill-taxonomy.js` 和 `src/team-ai-weights.js`，再在 `battle-engine.js` 之后、`app.js` 之前加载 `src/team-ai.js`。

## 郊野守军默认规则

郊野守军视为拥有全部资源，但每次换守军不会直接从全库最优解固定取队，而是先制造变化：

1. 从可用 5 星武将池中随机抽取 20 个候选。
2. AI 在候选里选择大营、中军、前锋 3 名武将。
3. 战法池默认只取 S 和 A 品质战法。
4. 每名武将配置 2 个战法，同一支队伍内战法不重复。
5. 不会给武将重复配置自己的自带战法。

玩家自动整备不再在 UI 层按星级或攻谋粗略截断到 24 个候选，而是把拥有武将交给 `recommendTeam()` 统一评分；AI 内部会做站位候选裁剪，避免全量三重枚举失控。

## 选将评分

AI 分站位评分，然后枚举大营、中军、前锋组合，选总分最高的一组。`cost` 只表示统率占用，不参与战力评分。

- 大营：重视攻击/谋略输出上限和攻击距离，短距离大营会被明显扣分。
- 中军：重视攻、谋、防、速的均衡，适合承接输出、辅助或控制。
- 前锋：重视防御和速度，其次才是攻击/谋略；低防御前锋会被惩罚。
- 星级只作为基础稀有度信号，权重低于站位职责、属性和距离。
- 阵容额外获得阵营协同、兵种协同、攻谋压力均衡、前锋承伤、大营射程和高速单位加分。
- 选将会读取候选武将的自带战法画像，把自带输出、治疗、控制、防御、增伤、射程支持、先手连击、禁疗压制、敌方减益和净化类价值折算进站位分与阵容分。
- 阵容分会检查队伍计划是否完整：至少要有可靠输出核心，前锋要能承压，队伍最好覆盖治疗/减伤、控制、开局增伤、敌方减益或先手连击等联动角色。
- 同名、同阵营、同兵种的重复官方卡视为同一身份，不会同时上阵。

`chooseLineup()` 会先从每个站位的高分候选和全局高分候选中取并集，再枚举组合。这保留了“某武将只适合特定站位”的机会，也避免候选池过大时浏览器卡顿。为保证换守军响应速度，单次推荐会预计算候选武将的站位分、总分、身份 key 和战斗画像；这些数据只在本次推荐内复用，不作为跨刷新缓存，不改变评分标准或候选规模。

## 战法评分

`chooseSkillsForHero()` 仍可单独对某个武将排序，`recommendTeam()` 实际组队时会调用 `chooseTeamSkills()` 做全队战法分配。它会从共享战法池里逐个选择当前边际收益最高的“武将 + 战法”组合，避免同一队伍重复堆太多同类治疗、辅助或控制，也避免某个武将独占全部高价值战法。

`src/skill-taxonomy.js` 负责从战法名称、类型、目标、效果、描述、距离和发动率中派生统一画像。画像包含 `profile`、`tags`、`mechanics`、`targetCount`、`damageRate`、`requiredSkillNames`、`trigger` 和 `reliability`，并在运行态通过 `enrichSkill()` 合并到战法对象。`team-ai.js` 优先读取这套结构化画像；如果模块没有加载，才回退到本模块内的旧文本推断。

战法评分核心依据是：

- 品质：S 高于 A，高于 B/C。
- 触发类型：指挥、被动稳定性更高；主动结合发动率；追击偏向攻击型和前锋/普攻型武将。
- 结构化画像：根据名称、类型、效果、描述和目标推断攻击、策略、控制、防御、辅助、治疗、射程、群体、增伤、先手/连击、禁疗、属性降低、净化、挑衅和策略溅射等标签。
- 属性适配：攻击/连击/反击型战法更看攻击，策略/控制/治疗/辅助/禁疗/减益/净化更看谋略，挑衅和承伤被动会额外看防御，先手连击会看速度。
- 期望收益：根据发动类型、发动率、准备回合、目标数量和描述里的伤害率，估算伤害期望；治疗按谋略和目标数估算治疗期望；控制、防御、增伤、射程支持、禁疗、减益、净化、先手连击和策略溅射也会折算为队伍价值。
- 站位适配：大营偏好远距输出、群体效果、策略溅射和敌方减益，中军偏好辅助/控制/净化/均衡输出，前锋偏好防御、控制、治疗、挑衅、连击和追击。
- 距离适配：大营要求更高射程，中军次之，前锋要求最低。
- 兵种适配：战法限定兵种与武将兵种匹配会加分，不匹配会被重罚，防止高品质但不可用的战法挤掉正确搭配。
- 队友联动：增伤/属性辅助/敌方减益会优先给有输出核心的队伍加分；治疗、净化和防御在前锋承压不足时更值钱；射程支持会照顾短距离大营/中军；连击、挑衅和策略溅射会结合武将攻击、防御、速度、谋略和队友输出压力估算收益。
- 条件联动：如果战法描述是“发动【某战法】时/后”才生效，`skill-taxonomy.js` 会把前置战法名提取到 `requiredSkillNames`。队伍里必须已经有对应前置战法或自带战法，否则会被重罚，避免把 `惊雷破阵` 这类战法配进没有【落雷】/【迷阵】的队伍。

官方战法如果没有手写战斗实现，仍会在 `app.js` 的官方战法行为推断中获得近似战斗效果，因此可以进入 AI 推荐池。S/A 战法复杂效果的当前战斗覆盖范围记录在 `docs/skill-coverage.md`；AI 的战法画像应随这些状态和专属战法补强同步更新。

## 扩展建议

- 如果要做玩家可见的“推荐理由”，优先在 `team-ai.js` 的评分结果里保留分项，而不是在 UI 中重新推断。
- 如果要做多套流派推荐，可以增加 profile 参数，例如 `burst`、`control`、`sustain`，调整站位权重、队伍协同和战法权重。
- 如果要做更强守军，可以提高候选池数量、只允许 S 级战法，或增加阵营/兵种协同权重。
- 如果要做更真实配将，应优先扩展 `skill-taxonomy.js` 的结构化画像规则，必要时再引入独立覆盖表，不要在 UI 或推荐结果里重复推断。

## 离线 benchmark

`src/team-ai-weights.js` 保存默认可调权重，`team-ai.js` 仍保持推荐 API 不变，并允许离线脚本通过 `weights` 参数传入候选权重。

离线评估入口：

```powershell
npm run benchmark:ai
```

它会启动本地 HTTP 页面，在真实浏览器运行态里复用官方数据归并、战法画像、AI 推荐和战斗引擎，从官方 5 星武将与 S/A 战法中抽样对战，输出胜率、平局率、平均回合、剩余兵力和样例队伍到 `reports/ai-benchmark-latest.json`。

自动调权入口：

```powershell
npm run benchmark:ai:tune
```

调权脚本使用固定 seed 的随机搜索生成候选权重，只有当候选综合分超过当前默认权重达到阈值时，才会直接写回 `src/team-ai-weights.js`。需要降低单次抽样过拟合时，使用多 seed 聚合评估：

```powershell
npm run benchmark:ai:tune:robust
```

也可以直接传入自定义 seed 组：

```powershell
node ./scripts/ai-benchmark.mjs --tune --seed-list=20260515,20260516,20260517,20260518
```

这条路径只用于离线迭代；不要把多轮模拟接入玩家点击“自动整备”的实时路径。
