# 0002 战报入口与全屏战报弹层

## 背景

主界面原先在右侧常驻显示战报详情，并且开战按钮按回合推进。随着战斗回放、战后统计和多封战报管理的需求增加，常驻战报面板会挤占战场空间，也无法承载类似游戏内“个人战报列表 -> 单封战报 -> 详情/统计”的层级。

## 决策

- 主界面不再直接展示战报详情，只保留顶部“战报”入口按钮。
- 点击“开战”后一次性模拟到战斗结束，主战场直接呈现结算后的兵力、胜负和“再战”状态。
- 每次完整战斗生成一封个人战报，未读战报数量显示在“战报”按钮徽标上。
- 点击战报入口打开覆盖全屏的战报弹层，先显示个人战报列表；选择某一封后进入单封战报。
- 单封战报内保留三个底部入口：`战报详情`、`统计`、`阵容详情`。当前实现中：
  - `战报详情` 按回合分组，并把同一武将连续产生的行动日志合并成行动组；左侧头像索引表示当前回合行动顺序，点击头像可跳转到对应行动组。
  - 行动组头部显示该武将本次记录后的兵力，伤害/治疗明细在目标名后补充目标剩余兵力，便于对照原版战报读法。
  - `统计` 拆成 `武将统计` 和 `战法统计` 两个视图。
  - `阵容详情` 使用战报快照渲染我方/敌方切换、站位、武将和战法配置。
- 单封战报的子视图中，右上角关闭按钮先返回“战斗地点”页；在“战斗地点”或个人战报列表页才关闭弹层。
- 战报弹层里的总兵力条和单武将兵力条都需要区分可战兵力与伤兵段，和主战场兵力条保持同一口径。

## 边界

- 战斗引擎仍只负责生成战斗状态、日志和结算结果，不接触 DOM。
- 战报列表、未读状态、全屏弹层和统计表都属于 `src/app.js` 的浏览器状态和渲染职责。
- 战斗状态事件仍进入战报；抽卡、拆解、自动整备等账号/系统事件继续进入系统消息。
- 战报持久化只保存可序列化快照，不保存战斗引擎运行时的循环引用或函数；日志可保存施放者/目标在该条日志后的兵力快照，用于战报 UI 展示，不反向驱动战斗结算。

## 验证

相关浏览器烟测覆盖：

- 开战后一次性结算并生成 1 封未读战报。
- 打开战报弹层并进入单封战报后，所选战报标记已读；如果本次只有一封未读战报，未读徽标清零。
- 战报详情中仍保留逐条日志 DOM，同时渲染回合行动组、左侧头像跳转和目标剩余兵力。
- 统计视图可以渲染武将统计和战法统计。
- 阵容详情视图可以切换我方/敌方，并从战报快照中显示三名武将与战法配置。
- 从统计或阵容详情点击关闭按钮会先返回战斗地点页，不会直接关回主界面。
- 战斗地点页的双方总兵力条与武将卡兵力条会显示伤兵分段。
